# v0.1.x Backport Release Guide

This document defines the fixed release guidance for the Filament 3 maintenance lane.

## Scope

- Applies only to the `filament/v3` branch.
- Covers `v0.1.x` patch/minor releases for Filament 3 consumers.
- Focuses on backporting compatible fixes and non-breaking maintenance updates from the `v0.2.x` line when feasible.

## Exclusions

- No Filament 4-only APIs or namespace migrations.
- No breaking changes to public package behavior in the `v0.1.x` lane.
- No `v0.1.x` tags from `main`; `main` remains the Filament 4 line (`v0.2.0+`).

## Decision table and CI matrix

- Backport decision table: [`README.md` -> "Filament 3 adaptation notes (`filament/v3` lane)"](/README.md#filament-3-adaptation-notes-filamentv3-lane)
- Test matrix (`v0.1.x`): [`README.md` -> "CI matrix (`v0.1.x`)"](/README.md#ci-matrix-v01x)

These two references are the source of truth for deciding if a change can be backported and how it must be validated.

## Upgrade and backport instructions

1. Start from `filament/v3` and identify the change candidate from `main` (`v0.2.x` line).
2. Apply only Filament 3-compatible adaptations (action classes, slug signatures, static type declarations, and UI parity).
3. Keep dependency constraints aligned with the `v0.1.x` lane policy.
4. Run the full test matrix jobs defined in `.github/workflows/tests.yml`.
5. Update `CHANGELOG.md` with a `v0.1.x`-lane note when publishing backport-facing behavior changes.
6. Tag from `filament/v3` using a `v0.1.x` version once CI is green.

## Known limitations

- Feature parity with `main` is best-effort; some `v0.2.x` work can be inapplicable to Filament 3.
- Backport work must preserve Filament 3 API contracts, which may require implementation differences from `main`.
- Release readiness for `v0.1.x` depends on green CI across all matrix combinations.
